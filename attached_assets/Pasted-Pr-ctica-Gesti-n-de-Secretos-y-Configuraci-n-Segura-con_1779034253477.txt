Práctica: Gestión de Secretos y Configuración Segura con simulación de escalamiento de privilegios

La práctica consistirá en desarrollar una API web básica con conexión a base de datos, autenticación de usuarios y control de roles. La aplicación será construida intencionalmente con algunos errores de seguridad para luego identificarlos, explotarlos de forma controlada en un entorno local y finalmente corregirlos aplicando buenas prácticas de ciberseguridad.

El objetivo principal será demostrar cómo una mala gestión de secretos, una configuración insegura y una validación deficiente de datos JSON pueden permitir riesgos como la exposición de credenciales o el escalamiento de privilegios.

1. Aplicación que vamos a crear

Vamos a desarrollar una API sencilla usando:

Node.js + Express
Base de datos MySQL o PostgreSQL
Archivo .env
Git y GitHub
Burp Suite
Visual Studio Code

La aplicación tendrá una gestión básica de usuarios con roles.

Los usuarios podrán:

Registrarse
Iniciar sesión
Consultar su perfil
Actualizar su información personal
Acceder a rutas según su rol

Existirán dos tipos de roles:

USUARIO
ADMIN

El usuario normal solo podrá acceder a funciones básicas, mientras que el administrador podrá acceder a rutas protegidas, como consultar la lista de usuarios registrados.

2. Vulnerabilidad principal que vamos a simular

La vulnerabilidad central será una manipulación del cuerpo JSON de una petición HTTP usando Burp Suite.

Técnicamente, esta vulnerabilidad se puede explicar como:

Mass Assignment por falta de validación de campos JSON

También la podemos presentar de forma más sencilla como:

Manipulación de JSON para escalamiento de privilegios

La idea es que el backend, en su versión insegura, permitirá actualizar datos del usuario sin verificar correctamente qué campos pueden o no pueden modificarse.

Por ejemplo, normalmente un usuario debería actualizar solo estos datos:

{
  "nombre": "Usuario Prueba",
  "correo": "usuario@test.com"
}

Pero mediante Burp Suite se interceptará la petición y se agregará un campo sensible:

{
  "nombre": "Usuario Prueba",
  "correo": "usuario@test.com",
  "rol": "ADMIN"
}

Si la aplicación está mal configurada, aceptará ese campo y cambiará el rol del usuario normal a administrador. Esto simula un escalamiento de privilegios.

3. Uso de Burp Suite

Burp Suite será la herramienta que nos permitirá capturar y modificar las peticiones enviadas desde el cliente hacia el servidor.

El flujo será este:

1. El usuario inicia sesión como usuario normal.
2. El usuario intenta actualizar su perfil.
3. Burp Suite intercepta la petición HTTP.
4. Se modifica manualmente el JSON enviado.
5. Se agrega el campo "rol": "ADMIN".
6. Se reenvía la petición modificada al servidor.
7. Se verifica si el rol del usuario cambió.
8. Se intenta acceder a una ruta protegida de administrador.

Con esto se demuestra que el problema no está en Burp Suite, sino en que el backend confía en datos que vienen del cliente.

4. Fase 1: Investigación inicial

En esta fase se documentará qué tecnologías se van a usar y por qué.

Se seleccionará una API REST sencilla porque permite trabajar fácilmente con peticiones HTTP, JSON, autenticación y conexión a base de datos.

También se investigarán temas como:

Gestión segura de secretos
Uso de variables de entorno
Configuración de archivos .env
Exclusión de secretos con .gitignore
Principio de mínimo privilegio
Control de roles
Validación de campos JSON
Uso de Burp Suite como proxy de interceptación

En el informe se explicará que la aplicación fue diseñada para contener fallos controlados con fines académicos, y que las pruebas se realizarán únicamente en entorno local.

5. Fase 2: Implementación insegura controlada

En esta fase se construirá la primera versión de la aplicación con errores intencionales.

La aplicación tendrá vulnerabilidades como:

Credenciales visibles en el código

La conexión a la base de datos estará escrita directamente en el código fuente.

Por ejemplo, de forma conceptual:

Usuario de base de datos: root
Contraseña: admin123
Nombre de base de datos: laboratorio_seguridad

Esto es inseguro porque cualquier persona con acceso al repositorio podría ver las credenciales.

Clave secreta del JWT expuesta

La aplicación usará autenticación con token, pero en la versión insegura la clave del token estará escrita directamente en el código.

Esto es peligroso porque esa clave sirve para firmar y validar tokens.

Variables sensibles sin protección

Datos como usuario, contraseña, host de base de datos, puerto y clave secreta estarán dentro del código, en vez de estar en un archivo .env.

Usuario de base de datos con privilegios excesivos

La aplicación usará inicialmente un usuario de base de datos con permisos administrativos innecesarios.

Esto representa un riesgo porque, si la aplicación es comprometida, el impacto sería mayor.

Falta de validación en datos JSON

Esta será la vulnerabilidad principal.

El backend aceptará campos enviados por el cliente sin revisar si son permitidos o no. Por eso, un usuario podrá intentar agregar campos sensibles como:

rol
estado
permisos
id_usuario

En nuestro caso se usará el campo:

rol

para simular el escalamiento de privilegios.

6. Fase 3: Ataque controlado con Burp Suite

En esta parte se realizará la prueba práctica.

Primero se creará un usuario normal con rol:

USUARIO

Luego ese usuario iniciará sesión y recibirá acceso a la aplicación.

Después intentará acceder a una ruta administrativa, por ejemplo:

GET /api/admin/users

El sistema debería responder:

{
  "mensaje": "Acceso denegado"
}

Luego se actualizará el perfil del usuario. La petición normal podría ser:

{
  "nombre": "Usuario Prueba",
  "correo": "usuario@test.com"
}

Pero antes de que llegue al servidor, Burp Suite interceptará la petición y se modificará así:

{
  "nombre": "Usuario Prueba",
  "correo": "usuario@test.com",
  "rol": "ADMIN"
}

Si la aplicación vulnerable acepta ese cambio, el usuario quedará con rol de administrador.

Finalmente, se intentará ingresar nuevamente a la ruta administrativa. Si ahora el acceso funciona, se habrá demostrado el escalamiento de privilegios.

7. Fase 4: Corrección segura y endurecimiento

Después de demostrar la vulnerabilidad, se corregirá la aplicación.

Uso de archivo .env

Las credenciales ya no estarán dentro del código.

Se creará un archivo .env con variables como:

DB_HOST=localhost
DB_USER=app_user
DB_PASSWORD=clave_segura
DB_NAME=laboratorio_seguridad
JWT_SECRET=clave_segura_larga
NODE_ENV=development

El código leerá esas variables desde el entorno.

Uso de .gitignore

Se configurará el archivo .gitignore para evitar subir secretos al repositorio.

Debe incluir:

.env
node_modules/

También se puede crear un archivo .env.example para mostrar qué variables necesita el proyecto sin exponer valores reales.

Ejemplo:

DB_HOST=
DB_USER=
DB_PASSWORD=
DB_NAME=
JWT_SECRET=
NODE_ENV=
Aplicación del mínimo privilegio en base de datos

Se creará un usuario específico para la aplicación.

Ese usuario tendrá solo permisos necesarios, por ejemplo:

SELECT
INSERT
UPDATE

Y no tendrá permisos peligrosos como:

DROP DATABASE
CREATE USER
GRANT
SUPERUSER

Esto reduce el impacto si ocurre una vulnerabilidad en la aplicación.

Validación estricta del JSON

La aplicación ya no aceptará todos los campos enviados por el usuario.

Solo permitirá modificar campos seguros como:

nombre
correo
password

Y bloqueará campos sensibles como:

rol
estado
permisos
id_usuario

Entonces, si alguien intenta enviar esto desde Burp Suite:

{
  "nombre": "Usuario Prueba",
  "correo": "usuario@test.com",
  "rol": "ADMIN"
}

El sistema deberá rechazarlo o ignorar el campo rol.

La respuesta podría ser:

{
  "mensaje": "Campo no permitido: rol"
}

O también:

{
  "mensaje": "Perfil actualizado correctamente"
}

pero manteniendo el rol original como:

USUARIO
Separación de ambientes

Se diferenciará la configuración de desarrollo y producción.

Por ejemplo:

.env.development
.env.production.example

La idea es demostrar que no se deben mezclar credenciales reales de producción con configuraciones locales de prueba.

8. Validación final

Después de aplicar las correcciones, se volverá a probar el mismo ataque con Burp Suite.

El flujo será:

1. Usuario normal inicia sesión.
2. Se captura la petición de actualización de perfil.
3. Se agrega "rol": "ADMIN" al JSON.
4. Se envía la petición modificada.
5. El backend rechaza o ignora el campo rol.
6. El usuario sigue siendo USUARIO.
7. El acceso a rutas administrativas continúa bloqueado.

Esto demostrará que la vulnerabilidad fue corregida correctamente.

10. Resultados esperados

Al final, la práctica debe demostrar dos momentos:

Antes de la corrección

La aplicación funciona, pero es insegura porque:

Expone credenciales.
Expone secretos.
Usa usuarios con permisos excesivos.
Permite modificar campos sensibles mediante JSON.
Permite escalamiento de privilegios.
Después de la corrección

La aplicación sigue funcionando, pero ahora es más segura porque:

Usa variables de entorno.
No sube secretos a GitHub.
Aplica mínimo privilegio en base de datos.
Valida los campos enviados por JSON.
Bloquea la modificación no autorizada del rol.
Separa configuración de desarrollo y producción.

12. Resumen final para explicar al profe

Bestie, podrías explicarlo así:

En esta práctica se desarrollará una API REST con autenticación, conexión a base de datos y control de roles. Inicialmente, la aplicación será implementada con errores de seguridad intencionales, como credenciales visibles en el código, secretos expuestos, usuario de base de datos con privilegios excesivos y falta de validación en los campos JSON enviados por el cliente.

La vulnerabilidad principal será una simulación de escalamiento de privilegios mediante Burp Suite. Para ello, se interceptará una petición de actualización de perfil y se modificará el cuerpo JSON agregando el campo rol: ADMIN. En la versión vulnerable, el sistema aceptará este cambio, permitiendo que un usuario normal obtenga permisos administrativos.

Posteriormente, se aplicarán medidas de seguridad como el uso de variables de entorno mediante .env, exclusión de secretos con .gitignore, creación de usuarios de base de datos con permisos mínimos, separación de ambientes y validación estricta de campos permitidos. Finalmente, se repetirá la prueba con Burp Suite para comprobar que el escalamiento de privilegios ya no es posible.

Esa explicación te queda súper completa para presentar la idea antes de implementar, bestie.
